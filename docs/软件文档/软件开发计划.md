## 2 项目概述

### 2.1 项目背景

随着舰船装备向机电液一体化、智能化方向发展，系统复杂度显著增加，传统的人工故障诊断方式已无法满足现代装备维护保障需求。为提升舰船装备的测试性水平和故障诊断效率，亟需开发一套集建模、分析、诊断于一体的智能化软件平台。该软件基于T语言（Test modeling Language）构建，采用信号-功能块图（FSB）建模方法，支持多领域（机械、电气、液压）系统的统一建模与故障诊断，为装备的测试性评价和快速故障定位提供技术支撑。

### 2.2 开发目标

开发测试性建模与交互式故障诊断一体化集成开发环境软件，实现以下目标：

1. **建模功能**：支持机电液多领域系统的可视化建模，提供器件库管理、连接关系定义、T语言模型编写等功能，构建完整准确的多领域耦合系统模型。

2. **测试性分析**：基于系统模型自动进行测试性评价，计算故障检测率（FDR）、故障隔离率（FIR）等关键指标，为测试方案优化提供数据支撑。

3. **诊断推理**：集成多信号图模型，支持基于约束满足的故障诊断推理，提供交互式诊断决策支持，实现快速准确定位故障根源。

4. **性能目标**：确保系统响应时间不超过10秒（测试性分析平均响应时间<10秒，交互式诊断实时响应时间<1秒），支持10000个以上可替换单元的可测性建模和分析。

### 2.3 软件产品描述

本软件为基于Qt框架开发的桌面应用程序，采用分层架构设计，包括数据对象层（DO）、业务逻辑层（BO）和用户界面层（Widget）。软件集成T语言建模引擎、多信号图诊断推理引擎、测试性分析引擎等核心组件，提供可视化建模界面、交互式诊断界面和测试性评价界面。主要功能模块包括：

- **系统集成建模模块**：支持机电液系统多层次建模，包括器件定义、端口连接、功能关系建模。
- **基本器件库管理模块**：提供10000+标准器件模型库，支持非标准器件扩展和T语言模型校验。
- **交互式诊断模块**：基于多信号图模型，提供动态诊断决策树生成和交互式故障定位。
- **测试性评价模块**：自动计算测试性指标，生成测试性分析报告，支持多种测试策略优化。

### 2.4 功能概览

根据需求规格说明，软件包含以下17项核心功能：

1. **系统集成-建模**：
   - 机械器件建模（T-Designer-001）：支持T语言模型编写、编译及维修信息配置。
   - 机械设备建模（T-Designer-002）：提供画布绘制、子块连接、器件属性配置功能。
   - 机电液设备集成建模（T-Designer-003）：实现跨领域器件选型、连接和属性统一配置。

2. **系统集成-测试性评价**：
   - 机械设备建模测试性评价（T-Designer-004）：自动统计系统信息并计算故障隔离率等指标。

3. **基本器件库管理**：
   - 电气器件管理（T-Designer-005~009）：提供本地物料库的查看、查找、编辑、删除、复制功能。
   - 基本器件库管理（T-Designer-010）：支持10000+器件模型浏览和管理。
   - 非标准器件建模（T-Designer-011）：支持自定义器件模型、T语言编写和功能块配置。
   - 非标准器件校验（T-Designer-012）：提供T语言模型语法校验功能。

4. **诊断与故障定位**：
   - 交互式诊断方案（T-Designer-013）：基于诊断树自动生成交互式诊断方案。
   - 液压泵站启动功能故障定位（T-Designer-014）：提供特定功能场景的故障定位。
   - 压力控制功能故障定位（T-Designer-015）：支持压力相关故障的诊断分析。
   - 测试性分析响应时间（T-Designer-016~017）：确保系统性能满足实时性要求。

## 4 项目组织与开发方式

### 4.1 开发组织架构

项目采用矩阵式组织结构，设立项目经理、系统架构师、开发团队、测试团队、质量保证组等角色。

**项目组织结构**：
- **项目经理**（1人）：负责项目整体规划、进度控制、资源协调、风险管理。
- **系统架构师**（1人）：负责系统架构设计、技术选型、关键技术攻关。
- **DO层开发组**（2-3人）：负责数据对象层设计实现，包括数据模型、持久化、序列化等。
- **BO层开发组**（3-4人）：负责业务逻辑层开发，包括建模引擎、诊断引擎、分析引擎等核心组件。
- **Widget层开发组**（2-3人）：负责用户界面层开发，包括UI设计、交互实现、模型视图等。
- **测试团队**（2-3人）：负责单元测试、集成测试、系统测试、性能测试等。
- **质量保证组**（1人）：负责过程质量控制、配置管理、评审组织等。
- **文档工程师**（1人）：负责技术文档编制、培训材料准备等。

### 4.3 开发方式

项目采用敏捷开发与瀑布模型相结合的混合开发模式。第一阶段采用瀑布模型完成需求分析和系统设计，确保架构合理性和设计完整性；第二、三阶段采用敏捷迭代开发，快速迭代实现功能模块，及时验证和调整；第四阶段采用V模型进行测试驱动开发，确保测试活动与开发活动同步开展。

**开发节奏**：每个迭代周期2-3周，包括需求分析、设计实现、测试验证、文档更新等环节。建立每日站会、每周例会、每月评审等沟通机制，确保信息及时传递和问题快速解决。

**协作方式**：采用Git版本控制，使用分支管理策略（develop分支、feature分支、release分支）。代码评审采用Pull Request机制，要求至少一人审查通过后才能合并。建立持续集成（CI）流水线，自动执行构建、测试、代码质量检查等任务。

## 5 生命周期、里程碑和交付物

### 5.1 软件生命周期模型

本项目采用V模型（V-Model）作为软件生命周期模型，将测试活动贯穿于开发全过程。V模型将需求分析、系统设计、详细设计、编码实现等左侧开发活动与对应的单元测试、集成测试、系统测试、验收测试等右侧测试活动相匹配，确保每个开发阶段都有相应的测试验证。

**V模型各阶段**：
- **需求分析阶段**：明确用户需求，制定需求规格说明，开展需求评审。
- **系统设计阶段**：设计系统架构、模块划分、接口定义，编写设计文档。
- **详细设计阶段**：设计模块内部结构、数据结构、算法流程，编制详细设计说明。
- **编码实现阶段**：按照设计文档编写代码，遵循编码规范，进行单元测试。
- **集成测试阶段**：集成各模块功能，验证模块间接口，进行集成测试。
- **系统测试阶段**：验证系统功能和性能，进行系统级测试和性能测试。
- **验收测试阶段**：用户参与验收，验证系统满足需求，签署验收报告。

### 6.3 进度监控机制

项目采用挣值管理（EVM）方法进行进度和成本控制。

**监控指标**：
- **计划值（PV）**：按计划应该完成工作的预算成本
- **实际成本（AC）**：已完成工作的实际成本
- **挣值（EV）**：已完成工作的预算成本
- **进度偏差（SV）**：SV = EV - PV，负值表示进度滞后
- **成本偏差（CV）**：CV = EV - AC，负值表示成本超支

**监控频率**：
- 每周统计一次进度数据
- 每月进行一次挣值分析
- 关键里程碑节点进行全面评估

**进度报告**：
- 每周提交《周进度报告》
- 每月提交《月进度报告》
- 里程碑节点提交《里程碑报告》

## 8 风险分析与管理

### 8.1 技术风险

**风险1：T语言建模引擎开发复杂度高**
- **风险描述**：T语言解析、语法检查、模型转换等技术实现难度大，可能影响开发进度和系统稳定性。
- **影响级别**：高
- **发生概率**：中等
- **应对策略**：
  - 提前进行技术预研，搭建原型验证关键技术可行性
  - 引入外部专家指导，学习业界成熟经验
  - 分阶段实现，优先实现核心功能，扩展功能迭代开发
  - 建立技术攻关小组，集中力量解决关键技术问题

**风险2：多信号图诊断算法性能瓶颈**
- **风险描述**：大规模系统的多信号图诊断推理计算量大，可能导致响应时间超标。
- **影响级别**：高
- **发生概率**：中等
- **应对策略**：
  - 采用图优化算法和并行计算技术提升诊断速度
  - 实现诊断结果的缓存机制，避免重复计算
  - 设计分级诊断策略，先粗定位再精确定位
  - 建立性能基准测试，持续监控和优化

**风险3：Qt Model/View架构迁移兼容性问题**
- **风险描述**：从QStandardItemModel向QAbstractItemModel迁移可能引入兼容性问题，影响现有功能。
- **影响级别**：中等
- **发生概率**：低
- **应对策略**：
  - 采用渐进式迁移策略，分模块逐步迁移
  - 建立完整的功能回归测试套件
  - 保留旧版实现作为备选方案
  - 加强代码审查，确保迁移质量

### 8.2 管理风险

**风险4：需求变更频繁**
- **风险描述**：用户需求不明确或频繁变更，导致开发计划调整和工作量增加。
- **影响级别**：中等
- **发生概率**：中等
- **应对策略**：
  - 加强需求调研和分析，充分理解用户真实需求
  - 建立需求变更控制流程，评估变更影响
  - 采用敏捷开发模式，快速响应需求变化
  - 与用户保持密切沟通，及时获取反馈

**风险5：关键人员流失**
- **风险描述**：核心开发人员或架构师离职，可能导致项目进度延误和技术断层。
- **影响级别**：高
- **发生概率**：低
- **应对策略**：
  - 建立知识管理和文档体系，降低人员依赖
  - 实施结对编程和交叉培训，培养多面手
  - 与核心人员签订保密和竞业限制协议
  - 建立人才储备和外部支持渠道

**风险6：进度控制失效**
- **风险描述**：任务分解不合理或进度监控不到位，导致项目延期。
- **影响级别**：中等
- **发生概率**：中等
- **应对策略**：
  - 采用WBS方法精细分解任务，制定合理工期
  - 建立挣值管理体系，实时监控进度和成本
  - 每周召开进度评审会，及时发现和解决问题
  - 设置进度缓冲时间，应对不可预见延迟

### 8.3 外部风险

**风险7：第三方组件依赖风险**
- **风险描述**：依赖的第三方组件（Qt、Z3等）出现兼容性问题或授权问题。
- **影响级别**：中等
- **发生概率**：低
- **应对策略**：
  - 选择成熟稳定的开源组件，降低依赖风险
  - 建立组件替代方案，预留切换时间
  - 定期评估组件维护状态和社区活跃度
  - 购买商业技术支持，获得专业服务保障

**风险8：验收标准不明确**
- **风险描述**：用户验收标准模糊或不切实际，导致验收困难。
- **影响级别**：中等
- **发生概率**：中等
- **应对策略**：
  - 在需求分析阶段明确验收标准和测试方法
  - 邀请用户参与阶段性评审，提前确认成果
  - 参考行业标准和最佳实践，制定合理指标
  - 保留验收测试的灵活性，适应实际需求

- 每周进行风险识别和评估
- 每个里程碑节点进行全面的风险回顾
- 建立风险登记册，记录所有已知风险

**风险应对流程**：

4. 监控风险状态
5. 风险关闭和经验总结

### 9.4 测试策略

**测试分层**：
1. **单元测试**：由开发人员执行，覆盖所有函数和类，语句覆盖率≥80%，分支覆盖率≥70%
2. **集成测试**：由测试人员执行，验证模块间接口和数据传递，接口覆盖率100%
3. **系统测试**：由测试团队执行，验证系统整体功能和性能，功能覆盖率100%
4. **验收测试**：由用户执行，验证系统满足业务需求，用户场景覆盖率100%

**测试类型**：
- **功能测试**：验证软件功能是否符合需求规格
- **性能测试**：验证系统响应时间、吞吐量、资源占用等性能指标
- **可靠性测试**：验证系统在长时间运行和异常情况下的稳定性
- **兼容性测试**：验证软件在不同操作系统、硬件配置下的兼容性
- **安全测试**：验证系统安全机制，扫描安全漏洞
- **易用性测试**：评估用户界面友好性和操作简便性

**测试工具**：
- **单元测试框架**：Qt Test Framework
- **性能测试工具**：QTest、Valgrind
- **静态代码分析**：Cppcheck、SonarQube
- **代码覆盖率**：gcov、lcov
- **缺陷管理工具**：Bugzilla、JIRA

- **高（High）**：核心功能不可用、性能严重不达标
- **中（Medium）**：一般功能缺陷、界面显示异常
- **低（Low）**：文本错误、提示信息不当

### 10.2 配置管理组织

**配置管理员**（1人）：
- 制定配置管理计划
- 管理配置项和基线
- 监督变更控制流程
- 进行配置审计
- 建立配置管理数据库

**变更控制委员会（CCB）**（3-5人）：
- 项目经理：负责变更决策
- 架构师：负责技术变更评审
- QA代表：负责质量影响评估
- 用户代表：负责业务影响评估

### 10.3 配置项识别

**代码类配置项**：
- 源代码文件（.cpp, .h）
- 资源文件（.ui, .qrc）
- 编译脚本（.pro, Makefile）
- 配置文件（.json, .xml）

**文档类配置项**：
- 需求规格说明
- 设计文档
- 测试文档
- 用户手册
- 项目计划文档

**数据类配置项**：
- 数据库脚本
- 测试数据
- 示例项目
- 基准测试数据

### 10.4 版本控制策略

**分支管理策略**：
- **master分支**：稳定版本，用于发布
- **develop分支**：开发主分支，集成各功能模块
- **feature分支**：功能分支，用于独立开发
- **release分支**：发布分支，用于发布准备
- **hotfix分支**：缺陷修复分支，用于紧急修复

**标签管理**：
- v1.0.0-Alpha：Alpha版本
- v1.0.0-Beta：Beta版本
- v1.0.0-RC：候选发布版本
- v1.0.0：正式发布版本

**版本号规范**：
采用语义化版本号（Semantic Versioning）：主版本号.次版本号.修订号
- 主版本号：不兼容的API修改
- 次版本号：向后兼容的功能新增
- 修订号：向后兼容的问题修正

### 10.5 变更控制流程

**变更申请流程**：
1. 提交变更申请单
2. CCB评审变更影响
3. 分配变更任务
4. 实施变更
5. 验证变更效果
6. 关闭变更申请

**变更分类**：
- **紧急变更**：影响系统正常运行的缺陷修复，立即处理
- **重要变更**：核心功能改进，需要CCB评审
- **一般变更**：次要功能完善，按计划处理
- **文档变更**：文档内容更新，流程简化

**变更控制工具**：
- 版本控制系统：Git
- 缺陷跟踪系统：JIRA
- 文档管理系统：SharePoint
- 变更管理流程：配置管理工具

### 10.6 基线管理

**基线类型**：
- **需求基线**：需求评审通过后的需求规格说明
- **设计基线**：设计评审通过后的设计文档
- **代码基线**：功能模块完成后冻结的代码
- **产品基线**：测试通过后准备发布的版本

**基线建立流程**：
1. 完成开发和测试
2. 提交评审申请
3. 组织评审会议
4. 根据评审意见修改
5. CCB批准建立基线
6. 标签和归档

### 10.7 配置审计

**审计类型**：
- **功能审计**：检查配置项功能完整性
- **物理审计**：检查配置项存在性和一致性
- **管理审计**：检查配置管理过程规范性

**审计频率**：
- 每周进行一次物理审计
- 每个里程碑进行一次功能审计
- 每季度进行一次管理审计

### 14.2 问题处理流程

**问题报告**：
- 任何人员均可发现问题
- 及时通过JIRA提交问题报告
- 详细描述问题现象和复现步骤
- 附上相关截图和日志

**问题分析**：
- 项目经理指派问题处理人
- 分析问题根因和影响范围
- 评估问题优先级和紧急程度
- 制定解决策略和时间计划

**问题解决**：
- 开发人员实施解决方案
- 测试人员验证解决效果
- QA人员确认问题关闭
- 更新问题处理记录

**问题跟踪**：
- 每日更新问题状态
- 每周评审问题处理进展
- 每月分析问题趋势
- 建立问题知识库

### 14.3 升级机制

**升级条件**：
- 问题超过处理时限
- 多次尝试无法解决
- 涉及多个模块或团队
- 影响项目关键路径
- 用户强烈投诉

**升级路径**：
- Level 1：问题处理人 → 模块负责人
- Level 2：模块负责人 → 技术负责人
- Level 3：技术负责人 → 项目经理
- Level 4：项目经理 → 部门领导
- Level 5：部门领导 → 公司高层

**升级处理**：
- 召开专题会议讨论
- 调配额外资源支持
- 调整项目计划
- 寻求外部专家帮助
- 制定应急方案

### 14.4 问题指标

**问题处理指标**：
- 问题响应时间：问题报告后4小时内响应
- 问题解决时间：一般问题24小时内解决，重大问题48小时内解决
- 问题关闭率：≥95%
- 问题重复率：≤5%
- 用户满意度：≥85%

- 每周问题数量统计
